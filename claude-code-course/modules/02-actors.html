认识主角们

想象一个电影剧组——导演、摄影师、演员、特技协调员和舞台监督。每个人都有明确的分工，大家齐心协力才能把戏演好。

主要角色

现在你已经了解了一条提示词的旅程，接下来让我们认识一下幕后团队。Claude Code 并不是一大坨代码——它由几个各司其职的专业模块组成，每个模块负责一件事。你可以把它们想象成电影剧组的成员，每个人都是自己领域的专家。

下面是主要组件的群聊，它们会轮流自我介绍。点击 下一条 来听听每个组件怎么说：

📺
REPL

大家好！我是 REPL——你输入 claude 后看到的终端界面。我负责处理你的输入、展示消息、管理提示框。你可以把我想象成摄影师——我把一切呈现给观众。

🎬
QueryEngine

我是 QueryEngine——导演。当用户发送提示词时，我负责协调整个流程。我决定何时调用 API、处理工具返回的结果，并管理对话历史。

🌐
API Service

我是 API 层——负责和云端的 Claude 大脑对话。我处理流式响应、管理身份认证、追踪 token 用量和费用。我就是那个登台表演的演员。

🔧
Tool System

我是 Tool System——特技协调员。当 Claude 需要读取文件、编写代码或执行命令时，我负责安全地完成这些操作。我会检查权限并执行具体动作。

🏗️
init()

我是 init()——舞台监督。在其他人开始工作之前，我负责加载配置、检查身份认证、初始化遥测，确保一切准备就绪。你永远看不到我，但没有我什么都跑不起来。

为什么有这么多文件？

每个文件只做一件事。QueryEngine.ts 负责协调，但从不接触 UI。REPL.tsx 负责展示，但从不直接调用 API。这种分离意味着你可以修改一个部分而不影响其他部分——这和电影剧组分工的道理一样，各有专长总比一个人包揽一切要好。

近距离观察每个角色

下面是每个主要组件的快速参考卡片。把它们记在心里——当出了问题时，知道该看哪个角色就是调试成功的 90%。

📺
REPL

screens/REPL.tsx — 终端界面。处理输入、展示消息、管理权限弹窗、键盘快捷键，甚至还有 vim 模式。超过 5000 行代码，是整个代码库中最大的文件。

🎬
QueryEngine

QueryEngine.ts — 会话协调器。管理对话历史、调用查询循环、追踪用量和费用、记录日志，并处理记忆加载。

🔄
query()

query.ts — 内层对话循环。构建 API 请求、处理流式响应、在上下文过长时执行压缩，并协调每轮对话中的工具执行。

🌐
API Service

services/api/claude.ts — 处理与 Claude 的所有通信。管理流式传输、重试机制、身份认证、模型选择和用量追踪。

🔧
Tool System

services/tools/ — 执行 Claude 请求的操作。足够聪明，能把只读工具并行执行以提高效率，同时让写入操作逐一执行以确保安全。

🏗️
init()

entrypoints/init.ts — 在应用启动前完成所有准备工作。配置校验、身份认证、代理设置、遥测初始化，以及优雅关闭的处理。

🎯
专业术语：关注点分离

这种模式——把一个大系统拆分成职责清晰的角色——叫做关注点分离（Separation of Concerns）。它是软件架构中最重要的原则。当你和 AI 编程工具对话时，可以这样说："把 API 逻辑放在服务层，别放在 UI 组件里"——听起来就很专业。
