深入理解 Claude Code 架构
01

什么是 Claude Code？

一台拥有超能力的对讲机——你对着它说话，一位天才助手回答你，而且它还能伸出手来帮你搞定事情。

终端里的 AI 助手

你打开终端，输入 claude，然后开始和一个能阅读你代码库、编辑文件、执行命令的 AI 聊天。但当你按下回车到看到回复之间，到底发生了什么？

Claude Code 是 Anthropic 打造的一款命令行 AI 编程助手。你用自然语言——就是日常说的大白话——提问或下达指令，它就能阅读你的代码、编辑文件、执行命令。就好比有一位资深开发者在和你结对编程，只不过他住在你的终端里。

💡
终端 vs. 浏览器

跟 ChatGPT（在浏览器中运行）不同，Claude Code 是一个 CLI 程序，直接在你的电脑上运行。这很关键，因为它可以直接访问你本地的文件和终端——再也不用在聊天窗口和编辑器之间来回复制粘贴代码了。

🧠
理解代码

读取你的文件，了解项目结构，理清各个模块之间的关系——就像一个刚把你整个代码库通读一遍的开发者。

✏️
编写代码

直接在你的电脑上编辑、创建和删除文件。再也不用从聊天窗口复制代码片段粘贴到编辑器里了。

⚡
执行命令

执行 Shell 命令、安装依赖、跑测试、检查结果——然后根据执行结果决定下一步该怎么做。

按下回车后发生了什么

当你输入一条提示词，比如 "修复 auth.ts 里的 bug"，然后按下回车，幕后会触发一整条事件链。让我们跟踪这条消息的完整旅程：

1
你输入提示词

你在终端里写下 "修复 auth.ts 里的 bug"，然后按下回车。

2
REPL 捕获你的输入

终端 UI（基于 React + Ink 构建）获取你的文本并准备进行处理。

3
上下文和你的消息打包在一起

你的提示词会被附加上额外的上下文：一个系统提示词（给 Claude 的指令）、过往对话的记忆，以及项目信息。

4
发送到 Claude 的 API

打包好的内容通过互联网发送到 Anthropic 的服务器，由 Claude AI 模型进行处理。

5
Claude 流式返回回复

回复一个 token 一个 token（逐词）地到达——而且可能包含使用工具的请求，比如编辑某个文件。

6
工具执行

如果 Claude 判断需要做点什么——比如读取文件或者应用一个编辑——工具系统就会在你的电脑上执行相应操作。

7
结果反馈回去

工具的执行结果（比如文件内容，或者"编辑已成功应用"）会被发送回 Claude，作为下一轮推理的输入。

8
最终回答

Claude 给出一个人类可读的回答："我在 auth.ts 第 42 行发现并修复了空值检查的 bug。"

✨
智能体循环（Agent Loop）

这种模式——AI 决定该做什么、执行操作、观察结果、然后再次决策的循环——被称为智能体循环。这是现代 AI 中最重要的概念之一。Claude Code 不只是在回答问题；它在一个持续的循环中进行推理、执行操作、不断调整。第 5–7 步可能会重复很多次，直到 Claude 给出最终答案。

专业术语：关注点分离

这种模式——把一个大系统拆分成职责清晰的角色——叫做关注点分离（Separation of Concerns）。它是软件架构中最重要的原则。当你和 AI 编程工具对话时，可以这样说："把 API 逻辑放在服务层，别放在 UI 组件里"——听起来就很专业。

03

对话循环

就像下一盘邮寄棋——你走一步，对手研究棋盘后回应，有时还会请裁判代为执行操作。如此你来我往，直到将杀为止。

心跳

query.ts 中的对话循环是 Claude Code 的心脏。每一次交互——从一个简单的提问到一次大规模的代码重构——都要经过这个长达 1,700 行的函数。它是让一切运转起来的引擎。

循环的核心是一个名为 query() 的异步生成器函数。它不会一次性算出完整答案再交给你，而是像电视直播一样，逐帧发送事件——文本片段、工具调用、状态更新——边生成边产出，而不是等整部电影渲染完毕。

query.ts export async function* query( params: QueryParams, ): AsyncGenerator< | StreamEvent | RequestStartEvent | Message | TombstoneMessage | ToolUseSummaryMessage, Terminal > { const consumedCommandUuids: string[] = [] const terminal = yield* queryLoop(params, consumedCommandUuids) for (const uuid of consumedCommandUuids) { notifyCommandLifecycle(uuid, 'completed') } return terminal }
通俗解读
第 1 行：定义一个名为 query 的函数，它会生成一个事件流。那个 * 让它成为生成器——可以把它想象成电视直播，一帧一帧地发送画面，而不是等整部电影拍完才给你看。
第 2–9 行：它可以产出不同类型的事件：流式数据块、新消息、工具使用摘要等等。每种类型代表对话回合中可能发生的不同事件。
第 10 行：跟踪在这个回合中消耗了哪些斜杠命令（比如 /compact 或 /clear）。
第 11 行：运行主对话循环。yield* 的意思是"把 queryLoop 的所有事件直接转发给监听者"——就像把一个直播信号接入另一个直播。
第 12–14 行：循环结束后，通知每个已消耗的命令执行成功。

这就是 Claude Code 代码库的真正入口。query() 函数只是一个轻量包装，它把所有繁重的工作都委托给了 queryLoop——也就是我们接下来要探索的真正引擎。

循环状态机

每次对话循环都需要在迭代之间记住一些东西——比如已经交换了哪些消息、当前是第几个回合、以及上下文窗口是否快满了。这个可变状态就是循环的"记忆"——一组随着对话推进而不断变化的变量。

在编程中，一个根据事件在一组预定义状态之间切换的系统被称为状态机。对话循环正是如此：它在"调用 API"、"处理响应"和"执行工具"等状态之间循环切换，一路携带着这个状态对象。

query.ts — State type type State = { messages: Message[] toolUseContext: ToolUseContext autoCompactTracking: AutoCompactTrackingState | undefined maxOutputTokensRecoveryCount: number hasAttemptedReactiveCompact: boolean maxOutputTokensOverride: number | undefined pendingToolUseSummary: Promise<ToolUseSummaryMessage | null> | undefined stopHookActive: boolean | undefined turnCount: number transition: Continue | undefined }
通俗解读
messages：完整的对话历史——你和 Claude 之间交换的每一条消息，加上工具返回的结果。它会随着每个回合不断增长。
toolUseContext：权限设置、可用工具和工作区信息，用于安全地执行工具调用。
autoCompactTracking：监控上下文（记忆）的使用量。当它变得太大时，会触发自动摘要压缩。
maxOutputTokensRecoveryCount：记录 Claude 达到输出上限后系统尝试要求它继续的次数。
hasAttemptedReactiveCompact：一个防止无限压缩循环的标志——每次问题只尝试压缩一次。
turnCount：循环当前处于对话的第几个回合。
transition：上一次迭代为什么继续（工具使用、压缩、恢复）——对调试很有用。
💡
为什么这很重要？

这个状态对象中的每个字段都源于生产环境中遇到的真实问题。例如，hasAttemptedReactiveCompact 之所以存在，是因为如果没有它，一段过长的对话会尝试压缩、失败、再次尝试、再次失败——永远循环下去。一个布尔标志就防止了无限循环。

循环的一次迭代

对话循环的每次迭代都遵循一个精确的流程。可以把它想象成邮寄棋局中的一个回合：收集棋盘状态，走一步棋，检查游戏是否继续。

1
构建 API 请求

把系统提示词、用户上下文、可用工具和完整的消息历史打包成一个请求，准备发送。

2
调用 Claude API

把所有内容发送到 Anthropic 的服务器，并开始流式接收响应。在 Claude 思考的过程中，连接保持打开。

3
处理流

收集到达的文本片段、工具使用块和思维块——同时转发到界面，让你实时看到文字逐步出现。

4
检查工具调用

响应完成后检查：Claude 是否请求了任何工具？如果是，准备执行这些工具。

5
执行工具

运行请求的工具——读取文件、编写代码、运行 shell 命令——并收集结果。

6
决定：继续还是停止？

如果有工具返回结果，就把它们加入消息历史，然后回到第 1 步。如果没有工具被请求，这个回合就完成了——交付最终答案。

智能体 vs. 聊天机器人

这种"决策-行动-观察"的循环正是智能体区别于简单聊天机器人的关键。聊天机器人给出一个回答就停了。而智能体会持续运转——做决策、执行操作、评估结果——直到任务完成。这就是为什么 Claude Code 能够处理像"重构这个模块并更新所有测试"这样复杂的多步任务。

当上下文太大时——压缩

对话中的每条消息都会占用 token——AI 的记忆单位。上下文窗口能容纳的 token 数量有硬性限制，而较长的编码会话会很快把它填满。想象你在一个只有 200 页的笔记本上写小说——最终会用光空间。

当这种情况发生时，Claude Code 会做一件聪明的事：它自动摘要对话中较早的部分以释放空间，就像把详细的会议记录浓缩成要点，这样你就有空间继续记录了。

🗜️
自动压缩

当 token 使用量超过阈值（上下文窗口的一定百分比）时，自动摘要较早的消息。这是主动式的——它在你撞墙之前就发生了，就像助手在你桌子溢出之前就帮你整理好了。

⚡
被动压缩

当 API 因为请求太长而拒绝时，紧急压缩对话并重试。这是安全网——它捕获自动压缩遗漏的情况，防止系统崩溃。

✂️
裁剪压缩

裁掉对话的中间部分，保留开头（初始设置和系统上下文）和结尾（最近的消息）。就像重新翻阅一本书的第一章和最后一章，回忆起前提和你读到哪里了。

04

工具系统

想象一间医院的手术室——外科医生说「手术刀」，护士先检查授权单，拿出正确的器械，安全地递过去。日常检查时，多位护士可以同时工作；做手术时，一次只能一位护士操作。

什么是工具？

Claude 不只会思考——它还能行动。当它决定读取文件、编辑代码或运行终端命令时，工具系统就是让这一切发生的核心机制。那么，工具到底是什么呢？

你可以把工具想象成一份 Claude 可以选择的「能力菜单」。每个工具都有一个名称（比如 ReadFile）、一段描述（让 Claude 知道什么时候该用它）、一个输入Schema（需要哪些参数），以及一个执行函数（它在你的机器上实际做什么）。Claude 阅读菜单，选择合适的工具，填入参数，然后工具系统替它执行操作。

📖
读取工具

读取文件、搜索代码库、列出目录。安全且快速——多个读取工具可以同时运行，互不冲突。

✏️
写入工具

编辑文件、创建文件、删除文件。需要权限，且一次只运行一个，防止出错。

⚡
Shell 工具

运行终端命令，比如 npm install、git commit 或 python test.py。功能强大但有潜在风险——始终需要权限才能执行。

🔌
MCP 工具

通过模型上下文协议连接外部服务——数据库、API、云服务等——全部通过标准化接口实现。

🤖
Agent 工具

创建子 Agent 来处理复杂的并行任务——就像主刀医生把一个子手术委托给专科医生。

🏥
手术室的比喻

把 Claude 想象成做决策的外科医生。工具系统就是护士团队：当医生要求一件器械时，护士会检查授权单（权限），选出正确的器械（找到工具），确认拿对了（检查 Schema），然后递过去。像查看病历这种常规操作，多位护士可以同时进行。而做手术——编辑文件、运行命令——一次只能一位护士操作，防止出错。

读取要快，写入要稳

这是让 Claude Code 既快速又安全的关键洞察：并非所有工具生来平等。读取文件不可能破坏任何东西——那为什么不同时读十个文件呢？但写入文件就不同了，如果两个写入操作碰撞，可能会造成损害。调度层通过一个巧妙的批处理策略解决了这个问题。

下面这个函数接收 Claude 想要发起的工具调用列表，并将它们分组成批次。并发安全的工具被分到一组；不安全的工具各自独占一个批次，串行执行（一次一个）。

通俗解读
接收一个工具调用列表，也就是 Claude 想要执行的操作，然后把它们分组成批次。
对每个工具调用，查找工具定义并验证其输入参数。这就是所谓的解析——确保参数与工具期望的格式匹配。
询问工具：「和其他工具一起运行安全吗？」只读工具说安全；写入工具说不安全。
如果这个工具是安全的，而且上一个批次也是安全的，就把它加入那个批次（归到一组并行执行）。
如果这个工具不安全，就开启一个新批次，只放这一个工具（它要单独运行）。
最终结果：连续的只读工具被打包到一起同时执行。写入工具始终独占一个批次，一次只运行一个。
⚡
批次 1 —— 并行（快速）

ReadFile("auth.ts") + ReadFile("utils.ts") + SearchFiles("login") —— 全部同时运行！

🔒
批次 2 —— 串行（安全）

EditFile("auth.ts") —— 在批次 1 完成后单独运行。

⚡
批次 3 —— 并行（快速）

这就是 Claude Code 代码库中真实的批处理函数。设计模式非常优雅：安全的工具被分组同时运行以提高速度，不安全的工具被隔离逐个运行以确保安全。没有锁、没有竞态条件——只有巧妙的调度策略。

权限关卡

在任何工具执行之前，它都必须通过权限关卡。这是 Claude Code 的安全网——防止 AI 在未经你同意的情况下删除你的文件、推送到 git 或运行危险命令的系统。

权限系统将其配置存储在一个上下文对象中。把它想象成医院的授权政策——一套规则，决定哪些操作可以自动进行，哪些需要患者（也就是你）的明确批准。

Tool.ts export type ToolPermissionContext = DeepImmutable<{ mode: PermissionMode additionalWorkingDirectories: Map<string, AdditionalWorkingDirectory> alwaysAllowRules: ToolPermissionRulesBySource alwaysDenyRules: ToolPermissionRulesBySource alwaysAskRules: ToolPermissionRulesBySource isBypassPermissionsModeAvailable: boolean shouldAvoidPermissionPrompts?: boolean >}
通俗解读
mode：当前的权限级别——「default」（危险操作每次都要询问）、「plan」（只读模式）或「bypass」（自动批准所有操作）。
additionalWorkingDirectories：除主项目目录外，Claude 被允许访问的额外文件夹。
alwaysAllowRules：一份白名单——匹配这些规则的工具会被自动批准，无需询问你。
alwaysDenyRules：一份黑名单——匹配这些规则的工具会被自动拦截，无论如何。
alwaysAskRules：无论其他设置如何，始终需要人工明确批准的工具。
isBypassPermissionsModeAvailable：用户是否启用了「信任一切」模式（即 --dangerously-skip-permissions 标志）。
shouldAvoidPermissionPrompts：针对没有 UI 来显示权限对话框的后台 Agent——它们会完全跳过交互式提示。

这套权限系统是 Claude Code 的安全网。它确保 AI 永远不会在未经你同意的情况下删除你的文件、推送到 git 或运行危险命令。当你构建 AI 驱动的工具时，一定要像这样构建权限关卡——这就是「好用的助手」和「定时炸弹」之间的区别。白名单/黑名单/始终询问这三者的组合，让你可以精细地控制 AI 能做什么、不能做什么。

启动与性能优化

想象一下 F1 赛车的进站维修团队——每一毫秒都至关重要。维修人员同时更换四个轮胎，在检查刹车的同时加油，甚至在赛车到达之前就已经准备好了零件。Claude Code 在启动时做的正是同样的事情。

三大性能策略

Claude Code 的启动时间不到 200 毫秒。对于这样一个复杂的程序——包含 AI API、插件系统、OAuth、遥测和完整的终端 UI——这个速度简直快得离谱。他们是怎么做到的呢？

在实际的软件工程中，性能优化并不是靠精妙的算法——而是靠不做不必要的工作。Claude Code 使用了三种策略：并行执行、懒加载（只在需要时才做），以及在编译时去除不需要的东西。

⚡
并行预加载

在程序一启动时就立刻发起耗时操作（钥匙串、MDM 设置、认证），然后在等待结果的同时继续做其他初始化工作。等到你真正需要结果时，它们早就准备好了。

🦥
懒加载

不要在启动时加载重量级模块，等到真正需要的时候再加载。OpenTelemetry（400KB）、gRPC（700KB）以及语音模式等功能都是按需加载的，而不是在启动时就全部加载。

✂️
编译时特性开关

打包器的特性开关系统在编译时就移除了整条代码路径。如果某个功能被禁用，它的代码在最终的 bundle 中根本不存在。

🏎️
进站维修的比喻

F1 赛车的维修团队不会一个一个地换轮胎——他们是四个轮胎同时换。他们不会等一个任务做完再开始下一个。Claude Code 运用的正是同样的思路：并行发起耗时任务、按需加载资源、让打包器剥离一切不需要的东西。最终结果？一个拥有数千项功能的程序，启动时间不到 200 毫秒。

并行预加载——进站维修团队

main.tsx 做的第一件事，就是在还没完成所有 import 之前，立刻发起耗时操作。注意代码是如何在 import 语句之间穿插函数调用的——这是故意的：每个耗时操作在后台开始运行的同时，剩余的代码继续加载。

通俗解读
在文件刚开始加载的那一刻就启动性能计时器。这样开发者可以精确测量启动的每个阶段花了多长时间。

🔑
预加载模式

尽早启动耗时操作，等到需要时再去获取结果。代码注释中提到，钥匙串读取大约需要 65ms，模块导入大约需要 135ms。通过让它们重叠执行，用户只需要等待较长的那个（135ms），而不是两者加起来（200ms）。当你让 AI 代理优化启动时间时，可以建议使用预加载模式。

❌
没有预加载（顺序执行）

钥匙串（65ms）→ 然后 MDM（35ms）→ 然后模块导入（135ms）= 总计 235ms。每个任务都要等上一个完成后才开始。

✅
使用预加载（并行执行）

钥匙串 + MDM + 模块导入同时启动 = 总计约 135ms。你只需要等最慢的那个，速度提升将近 2 倍！

懒加载——按需付费

重量级模块不会在启动时加载——它们是在真正需要时才被动态导入的。如果用户根本不会用到需要 gRPC 的功能，何必加载 700KB 的 gRPC 代码呢？

Memoize（记忆化）模式

记忆化模式在性能敏感的代码中非常常见。可以把它想象成做完一道难题后把答案写在便利贴上——下次有人再问，你直接看便利贴，而不用重新解题。在 AI 工具中，记忆化被广泛使用来避免重复计算。

编译时特性开关

Claude Code 使用了一种叫做死代码消除的强大技术。打包器在编译时求值 feature() 标志，然后直接把被禁用的代码从最终文件中移除。这不是运行时的检查——如果该标志关闭，代码在你的安装中物理上就不存在。

普通的 if/else 检查发生在运行时——两个分支的代码都存在于最终文件中，占用空间并拖慢加载速度。而使用编译时特性开关，打包器看到条件永远为 false，就会删除整个分支，最终文件在物理上变得更小。这就好比在一个房间门口挂个"关闭"的牌子，和直接把那个房间拆掉之间的区别。

Print 模式的捷径

Claude Code 对它的 --print（无头/非交互式）模式有一个巧妙的优化。当你运行 claude -p "explain this function" 时，你不需要完整的交互体验——你只需要一个答案。那为什么要浪费时间去设置你根本用不到的功能呢？

2

不注册 doctor、mcp、config、api-key 等等。节省数十毫秒。

3
直接进入构建查询

跳过其他步骤，直奔核心：构建提示词、调用 API、流式输出响应、打印结果。

核心要点

Claude Code 之所以能让一个极其复杂的程序在 200 毫秒内启动，靠的是一个原则：绝不做不需要做的工作。预加载让耗时操作重叠执行，这样你只需要等最长的那个而不是全部加起来。记忆化确保昂贵的函数只运行一次。编译时特性开关从物理上移除未使用的代码。Print 模式跳过一切非必要的步骤。这些并不是什么高深的技巧——它们是实用的工程模式，你可以应用到你开发的任何软件中，也可以让 AI 来帮你实现。

常见故障排查

现在你已经能看到整张地图了，让我们来聊聊出问题时会发生什么。以下是用户最常遇到的五个问题——以及在架构中该到哪里去排查：

1
Claude 没有响应

检查 API 层（services/api/）。可能是认证问题、速率限制或网络故障。这属于区域 3——你的机器和云端 Claude 之间的连接。

2
Claude 有回复但没有做出更改

检查工具系统（services/tools/）。可能是权限阻止了执行。Claude 知道该做什么，但安全门控拦住了它——看看区域 4。

3
Claude 似乎忘记了之前的上下文

检查上下文压缩（services/compact/）。对话可能被自动摘要了，丢失了具体细节。这是因为上下文窗口空间不够了。

4
启动缓慢或崩溃

检查 init()（entrypoints/init.ts）。配置解析错误或代理问题是常见原因。这属于区域 5——其他一切都依赖的地基。

5
Claude 陷入循环，没有进展

检查查询循环（query.ts）。可能需要调整最大轮次限制或 token 预算。Claude 可能在重复同一个操作，或者对话状态卡在了一个死循环里。

值得了解的设计模式

在整个课程中，你反复看到了相同的工程模式。这些并非 Claude Code 独有——它们是整个软件行业久经考验的策略。了解它们，你就能读懂专业软件是怎么构建的。

🔁
Agent Loop（智能体循环）

AI 做出决策、执行动作、观察结果、然后重复。这是所有 AI 编程工具背后的核心模式，也是让智能体区别于简单聊天机器人的关键。

⚡
Parallel Prefetch（并行预取）

提前启动耗时操作，稍后再收集结果。用于启动优化——同时发起网络请求和文件读取，这样就没有任务需要排队等待。

🧠
Memoization（记忆化）

计算一次，永远复用。用于 init() 和许多内部缓存。既然答案没变，为什么要重新计算一遍呢？

✂️
Feature Flags（功能标志）

在编译时开关功能，彻底去除未使用的代码。不会发布任何死代码——如果功能门控关闭了，那段代码在构建中就不存在。

🔒
Permission Gates（权限门控）

操作之前，必先请求许可。这是 AI 工具的安全原则。每个破坏性操作都必须通过检查，让人类拥有最终决定权。

🗜️
Context Compaction（上下文压缩）

摘要旧对话以适应 AI 记忆限制。用细节换容量——就像把会议记录浓缩成要点，这样才能继续讨论。

📺
Terminal React（终端 React）

用 React（一个 Web UI 框架）在终端中渲染丰富的界面。组件化、状态管理、声明式渲染——只不过是在命令行里而不是浏览器中。

🔄
Async Generators（异步生成器）

逐段传输数据，而不是等到全部完成。用于对话循环——你能实时看到 Claude 思考时逐字出现的文字。
